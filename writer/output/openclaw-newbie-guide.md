## 二、OpenClaw 是什么？

用一句话说清楚：

> **OpenClaw 是一个自托管的 AI 网关，把你的聊天软件变成 AI 助理的对话框。**

### 核心架构

```
┌─────────────────┐     ┌──────────────┐     ┌─────────────────┐
│   聊天客户端     │     │   Gateway    │     │   AI Agent      │
│  WhatsApp       │────→│   网关服务    │────→│   Pi/Claude     │
│  Telegram       │     │   路由/会话   │     │   工作空间      │
│  Discord/飞书   │     │   工具链      │     │   记忆系统      │
└─────────────────┘     └──────────────┘     └─────────────────┘
```

### 为什么需要它？

想象这个场景：

你在 WhatsApp 上给一个号码发消息，对面是一个 24 小时待命的 AI 助理。它可以：
- 帮你查日历、回邮件
- 写代码、改文档
- 管理待办、生成日报
- 甚至控制你的智能家居

**关键特性：**
- **自托管**：跑在你的机器上，你的数据你做主
- **多通道**：一个网关同时服务 WhatsApp、Telegram、Discord、飞书
- **多 Agent 路由**：不同聊天/群组可以路由到不同的 AI 人格
- **会话隔离**：每个对话有独立的记忆和上下文
- **工具链扩展**：通过 Skills 系统扩展能力

## 三、核心概念：理解 OpenClaw 的"世界观"

### 1. Gateway（网关）

Gateway 是 OpenClaw 的核心服务，一个持续运行的进程。

**它负责：**
- 连接各种聊天平台（Channels）
- 管理会话状态（Sessions）
- 路由消息到正确的 Agent
- 处理媒体文件（图片、音频、文档）

Gateway 是"单点真相"——所有会话、路由、连接状态都在这里。

### 2. Agent（智能体）

每个 Agent 是一个独立的"大脑"，拥有自己的：

**工作空间（Workspace）**
```
~/.openclaw/workspace/
├── SOUL.md          # 人格定义（我是谁）
├── AGENTS.md        # 操作规范（怎么工作）
├── USER.md          # 用户上下文（服务谁）
├── MEMORY.md        # 长期记忆（可选）
└── memory/
    └── YYYY-MM-DD.md  # 每日日志
```

**会话存储**
```
~/.openclaw/agents/<agentId>/sessions/
├── sessions.json    # 会话元数据（token 用量、最后路由等）
└── <SessionId>.jsonl  # 会话历史
```

**关键点：**
- 每个 Agent 可以配置不同的 API Key 和模型
- 工作空间可以 Git 化备份
- 发送 `/new` 或 `/reset` 可以重置会话

### 3. Channel（通道）

已支持的通道：

| 通道 | 状态 | 说明 |
|------|------|------|
| WhatsApp | ✅ | 通过 WhatsApp Web (Baileys) |
| Telegram | ✅ | Bot API (grammY) |
| Discord | ✅ | Bot API |
| 飞书 | ✅ | 企业应用 |
| iMessage | ✅ | macOS 本地 imsg CLI |
| Mattermost | 🔌 | 插件扩展 |

### 4. Binding（路由绑定）

多 Agent 场景下，消息如何路由到正确的 Agent？

**路由优先级（精确匹配优先）：**
1. 精确匹配（特定群组/用户 ID）
2. 父级继承（线程继承）
3. 角色匹配（Discord roles）
4. 服务器匹配（Discord guilds）
5. 账号匹配（accountId）
6. 通道级匹配（channel: "whatsapp"）
7. 默认 fallback

**示例逻辑：**
```
工作群消息 → 精确匹配 → 工作 Agent
私人 WhatsApp → 通道匹配 → 生活 Agent
Telegram 消息 → 通道匹配 → 深度工作 Agent
```

## 四、使用场景：怎么"用"，不是怎么"配"

### 场景 1：个人助理——24 小时待命的对话框

**典型用法：**
- 专用一个 WhatsApp 号码作为"助理入口"
- 从你的主力手机发消息给它
- 它帮你查日历、回邮件、写代码、记笔记

**为什么有效：**
- 不用打开新 App，就在聊天界面
- 有完整的对话历史，上下文连续
- 可以发图片、语音、文档

### 场景 2：多 Agent 协作——工作/生活分离

**问题：** 你不想让"写代码的 Agent"知道你家里的琐事，也不想让"聊天 Agent"访问你的工作代码。

**解法：** 两个 Agent，两个工作空间，完全隔离。

```
工作 Agent
├── 模型：Claude Opus（强，但贵）
├── 工具：代码执行、文件读写
├── 入口：Telegram
└── 记忆：项目文档、技术笔记

生活 Agent
├── 模型：Claude Sonnet（快，便宜）
├── 工具：日历、邮件、待办
├── 入口：WhatsApp
└── 记忆：家庭日程、购物清单
```

**实际体验：**
- 在 Telegram 上问技术问题 → 工作 Agent 回答
- 在 WhatsApp 上问"今晚吃什么" → 生活 Agent 回答
- 两边互不知道对方的存在

### 场景 3：群聊机器人——@提及触发

**典型用法：**
- 在项目群里，@笔杆子 让它写日报
- 在技术群里，@代码极客 让它 review 代码
- 在家人群里，@家庭助理 让它查天气

**触发规则：**
- 群消息默认静默（不主动回复）
- 只有被@时才响应
- 可以配置多个触发词（@openclaw、@助理、@bot）

### 场景 4：心跳任务——主动汇报

**什么是心跳？**
OpenClaw 可以定期（比如每 30 分钟）主动唤醒 Agent，让它检查任务并汇报。

**典型心跳任务：**
- 检查未读邮件
- 查看日历（未来 24 小时）
- 生成项目日报（18:30 自动推送）
- 检查 GitHub 通知

**工作机制：**
1. 心跳触发，Agent 读取 `HEARTBEAT.md`
2. 执行检查任务
3. 如有重要发现，主动推送消息
4. 如无异常，回复 `HEARTBEAT_OK`（自动静默，不打扰你）

## 五、安全哲学：为什么"自托管"重要

### 自托管的意义

当你用商业 AI 服务时：
- 你的对话数据存在对方服务器
- 对方可以训练模型（可能泄露你的信息）
- 服务可能关停、涨价、改条款

当你自托管 OpenClaw 时：
- 所有数据在你自己的机器上
- API 调用直接走官方（Anthropic/OpenAI）
- 你控制一切：权限、工具、会话

### 安全配置理念

**1. 白名单原则**
只允许信任的号码/用户发消息给 Agent。

**2. 沙箱隔离**
对不信任的场景（比如家庭群），限制 Agent 的工具权限：
- 允许：读文件、执行命令
- 禁止：写文件、访问浏览器

**3. 会话隔离**
每个对话有独立的会话存储，互不干扰。

**4. 群聊 gating**
群消息默认不响应，只有被@时才触发。
